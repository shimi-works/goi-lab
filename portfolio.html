Case Study — Web App / LLM

語彙ラボ

日本語の語彙力を「暇つぶし」で育てる学習ツール。
調べた語が自分の語彙帳に貯まり、忘却曲線に沿って復習が回りだす。

制作 / 個人 形態 / 単一HTML・ノーサーバー LLM / Google Gemini（無料枠） 保存 / localStorage 2026

これは何か

Overview

知らない語を調べると、意味・類義語・対義語・例文・使われる場面までを言語モデルが生成し、 そのまま自分の語彙帳に蓄積される。貯めた語は 4段階の認知度で管理され、クイズと忘却曲線で復習が回る。 「調べる → 貯める → 忘れる前に出会い直す」を一つのアプリで完結させた。

青空文庫リーダーやレシピ集約アプリで培った「テキストを貼る → 必要な情報を抽出する」手法を、 語彙学習に転用したもの。単発の便利ツールではなく、使うほど自分専用になる設計を狙った。

設計思想

Design Philosophy

「便利」を正面から避けた。 意味を返すだけのツールは ChatGPT に聞けば済み、分かりやすい便利機能は誰かに先を越される。 そこで「対話型AIでは代替できないこと」——自分のデータが貯まり続ける・状態を覚えている・忘れた頃に出会い直させる ——だけに機能を寄せた。価値の源泉を「賢さ」ではなく「あなた専用に育つこと」に置いている。

学習を"体験"として設計した。 語彙は覚えて終わりではなく忘れるもの。だから知識量ではなく継続を設計対象にした。 認知度タグ・忘却曲線・連続学習日数・成長グラフは、どれも「今日また開く理由」を作るための仕掛けとして一貫している。

制約を割り切りではなく設計に変えた。 お金をかけない・サーバーを持たない・個人で使う——この3制約を前提に、 鍵は各自が持ち込み（BYOK）、データは端末に置き、通信は無料枠のみ、という構成に落とした。 制約から出発したことで、逆に「配っても自分の負担がゼロ」という強みが生まれた。

技術構成

Architecture

配信形態
単一HTMLファイル。ビルド工程・フレームワーク・依存パッケージなし。ダブルクリックでも静的ホスティングでも動く。
永続化
localStorage に全データ（語彙・豆知識・認知度・学習履歴・設定・結果キャッシュ）。JSON書き出し／読み込みでバックアップと端末間移行。
AI 呼び出し
ブラウザから Gemini API を直接叩く。APIキーはユーザーが自分のものを入力（BYOK）。構造化出力（responseSchema）で JSON を安定生成。
グラフ描画
外部ライブラリを使わず SVG を手組み。オフライン動作・CSP制約下でも安全。

主要な技術的判断

Key Decisions — 課題 / 判断 / 理由

LLM プロバイダの選定

課題
高品質な生成を使いたいが、継続利用が前提で、無料かつアカウントが消えない土台が要る。
判断
当初の候補（Claude）は無料枠がなく、手持ちのアカウントも一時的だったため断念。Gemini の無料枠に切り替え、自分のGoogleアカウントで永続運用する構成にした。
理由
「無料・永続・各自の鍵」を満たすのはこれだけ。品質より先に、続けられる土台を優先した。

無料枠の枯渇に耐える設計

課題
無料枠は分・日単位の上限があり、少し使うとすぐ止まる。止まると学習が途切れる。
判断
3つの多層対策——①上限(429)で自動的に別モデルへ巡回（無料枠はモデルごとに別なので実質約3倍）②混雑(503)は待って自動リトライ③一度調べた語はキャッシュして再取得しない。
理由
ユーザーに設定を触らせず、失敗を"見えないところで吸収"する。継続を止めない仕組みそのものが機能になる。

忘却曲線ベースの出題

課題
ランダム出題では、覚えた語も知らない語も同じ頻度で出て非効率。
判断
各語に復習間隔（1・3・7・16・35・70日）を持たせ、正解で先送り／不正解でリセットする軽量な間隔反復を実装。出題は復習期限が来た語 ＞ まだ覚えていない語 ＞ 全体の順で選ぶ。
理由
「忘れかけたタイミングで出会い直す」を自動化。少ない問題数で定着率を上げる。

認知度と成長の可視化

課題
"知ってる"と"覚えた"は違う。前者は自己申告、後者は学習の成果。混ぜると成長が見えない。
判断
認知度を知らない／あやふや／知ってる／覚えた！の4段階に分離。登録時に選べるのは前3つで、覚えた！はクイズで正解して初めて到達する状態にした。到達時刻を記録し、覚えた数の推移をグラフ化。
理由
「学習して初めて上がる」段階を作ることで、右肩上がりの線が努力の証明になる。

非破壊な段階的開発

課題
実際に使う中で要望が次々出る。触るたびに動いている機能を壊すリスクがある。
判断
新機能は原則追加のみ（既存の関数・データ構造・画面に手を入れない）。新フィールドは既定値で吸収し、古い保存データや旧キャッシュを壊さない後方互換を徹底した。
理由
個人開発でもテストの代わりに"壊さない構造"で守る。実利用しながら育てるための前提。

機能マップ

Features

制約と割り切り

Honest Limitations

承知の上で選んだトレードオフ

  • 毎日のプッシュ通知は載せていない。 ブラウザWebはアプリを閉じている間に通知を送れない（特にiOS）。確実にやるならネイティブ化が必要——ここは「開いたら連続日数が見える」設計で継続動機を代替した。
  • APIキーは端末内に保存＝個人利用が前提。 不特定多数への公開にはプロキシ（サーバー）が要る。一方、ファイルを渡す形なら各自が自分の鍵を入れるので、無料のまま安全に配れる。
  • 無料枠の上限は残る。 多層対策で緩和はしたが、根本解決には少額課金が要る。個人の学習用途では無料の範囲で成立する、という判断。

この作品の位置づけ

Portfolio Axis

わたしの制作物に共通する軸は、
「学習と創作の体験を設計すること」。

青空文庫を読みやすくするリーダー、レシピを集約して費用と時間を見せるツール、 猫視点でストリートビューを歩く体験、傾けて絵が変わるレンチキュラー—— テーマは違っても、狙いはいつも「そのままでは続かない・触りにくいものを、続けたくなる体験に変える」こと。 語彙ラボは、その軸を"毎日ひらく学習ツール"という形で最も素直に体現した一つ。
